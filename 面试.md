# Niche Review 项目面试题整理

这份文档用于面试前复习。内容围绕本项目已经实现的功能展开，重点整理面试中常问的业务设计、并发问题、Redis/RocketMQ/Redisson 原理，以及项目中对应的解决方案。

## 1. 项目整体介绍

### 1.1 你这个项目是做什么的？

这是一个本地生活点评类项目，类似简化版大众点评。项目包含用户登录、商户查询、附近商户、博客探店、点赞、关注、Feed 流、签到、优惠券秒杀、异步下单、超时关单等功能。

技术栈包括：

- Spring Boot
- MyBatis-Plus
- MySQL
- Redis
- Redisson
- RocketMQ
- Lua
- JMeter
- Nginx

项目重点不是普通 CRUD，而是围绕 Redis 和高并发秒杀场景做了多种改造：

- Redis 缓存商户信息
- 布隆过滤器防止缓存穿透
- Redis GEO 实现附近商户
- Bitmap 实现签到
- ZSet 实现关注 Feed 流
- Lua 实现秒杀库存原子预扣
- RocketMQ 实现异步下单
- RocketMQ 延迟消息实现超时关单
- Redisson 分布式锁解决并发问题
- AOP + Redis ZSet + Lua 实现滑动窗口限流

### 1.2 这个项目和普通黑马点评项目相比有什么改进？

普通教程项目主要围绕 Redis 基础数据结构和简单秒杀展开。当前项目额外做了这些增强：

- 使用 Redisson 布隆过滤器降低缓存穿透风险
- 使用滑动窗口限流保护秒杀接口
- 使用 RocketMQ 替代 Redis Stream 实现异步下单
- 使用 RocketMQ 延迟消息实现超时自动关单
- 引入支付回调模拟逻辑
- 使用订单维度分布式锁解决支付回调和超时关单并发冲突
- 在 MQ 消费端增加幂等判断和失败重试策略
- 使用数据库唯一索引兜底保证一人一单

## 2. 登录业务

### 2.1 登录流程是怎样的？

流程：

```text
用户提交手机号和验证码
        |
后端校验验证码
        |
查询用户是否存在
        |
不存在则创建用户
        |
生成随机 token
        |
把 UserDTO 写入 Redis Hash
        |
返回 token 给前端
        |
前端保存 token，后续请求放入 Authorization 请求头
```

Redis Key：

```text
login:token:{token}
```

Value 是 Hash，保存用户的基本信息，例如：

```text
id
nickName
icon
```

### 2.2 为什么不用 Session，而是用 Redis + Token？

传统 Session 存在服务端内存里，单机使用没有问题。但如果后端部署多台机器，请求可能被负载均衡到不同服务器，Session 就会出现共享问题。

使用 Redis + Token 的好处：

- 登录状态统一存储在 Redis
- 多台后端服务都可以访问同一份登录态
- Token 可以跨服务传递
- 更适合分布式部署

### 2.3 ThreadLocal 在项目中有什么作用？

项目中 `UserHolder` 使用 ThreadLocal 保存当前请求的登录用户。

请求进入后：

```text
RefreshTokenInterceptor
        |
根据 token 查询 Redis 用户信息
        |
转换成 UserDTO
        |
保存到 UserHolder(ThreadLocal)
```

后续业务代码可以直接获取：

```java
UserHolder.getUser()
```

不用每个方法都传 userId。

### 2.4 ThreadLocal 为什么要 remove？

Tomcat 使用线程池处理请求，线程会被复用。

如果请求结束后不清理 ThreadLocal，可能出现：

```text
上一个用户的信息残留在线程里
下一个请求复用了同一个线程
导致拿到错误用户
```

所以拦截器的 `afterCompletion` 中要执行：

```java
UserHolder.removeUser();
```

## 3. 缓存相关问题

### 3.1 商户缓存是怎么做的？

商户详情查询时，先查 Redis：

```text
cache:shop:{shopId}
```

如果 Redis 有数据，直接返回。

如果 Redis 没有数据，再查 MySQL，并把查询结果写入 Redis。

流程：

```text
查询商户
        |
先查 Redis
        |
命中则返回
        |
未命中则查 MySQL
        |
MySQL 有数据则写 Redis
        |
返回结果
```

### 3.2 什么是缓存穿透？项目怎么解决？

缓存穿透是指：用户查询一个数据库根本不存在的数据。

例如：

```text
shopId = -1
shopId = 999999999
```

这类数据 Redis 没有，MySQL 也没有。如果大量请求打过来，每次都会绕过 Redis 去查 MySQL，导致数据库压力很大。

项目中用了两种方式：

1. 空值缓存
2. 布隆过滤器

空值缓存：

```text
MySQL 查不到数据
        |
Redis 写入空字符串
        |
设置较短 TTL
```

下次再查同一个不存在的 ID 时，Redis 能命中空值，直接返回不存在。

布隆过滤器：

```text
项目启动时把合法 shopId 加入布隆过滤器
        |
查询商户前先判断 shopId 是否可能存在
        |
如果布隆过滤器判断一定不存在，直接返回
```

### 3.3 布隆过滤器的原理是什么？

布隆过滤器底层是一个很长的二进制数组。

加入数据时：

```text
一个值经过多个 hash 函数
得到多个数组下标
把这些位置设置为 1
```

判断数据是否存在时：

```text
同样经过多个 hash 函数
检查对应位置是否都为 1
```

如果有任意一个位置是 0：

```text
这个数据一定不存在
```

如果所有位置都是 1：

```text
这个数据可能存在
```

所以布隆过滤器有两个特点：

- 判断不存在是准确的
- 判断存在可能误判

### 3.4 为什么布隆过滤器会误判？

因为不同的数据经过 hash 后，可能会把相同的位置设置为 1。

某个不存在的数据查询时，刚好对应的几个位置都被其他数据设置成 1，就会被误判为可能存在。

但是布隆过滤器不会误判不存在：

```text
只要有一个位置是 0，就说明这个数据一定没有加入过
```

### 3.5 为什么使用 Redisson 的布隆过滤器？

自己实现布隆过滤器需要处理：

- 位数组存储
- 多个 hash 函数
- 误判率计算
- 容量初始化
- Redis 操作封装

Redisson 已经封装好了这些能力：

```java
RBloomFilter<String> bloomFilter = redissonClient.getBloomFilter("bf:shop:id");
```

使用简单，适合学习项目快速接入。

### 3.6 Redisson 布隆过滤器为什么 Redis 里有两个 Key？

Redisson 布隆过滤器通常会有：

```text
bf:shop:id
bf:shop:id:config
```

其中：

- `bf:shop:id` 保存位图数据
- `bf:shop:id:config` 保存配置，例如预计插入数量、误判率、hash 次数等

这是 Redisson 的内部实现方式，不是项目写错了。

## 4. Redis GEO 附近商户

### 4.1 附近商户是怎么实现的？

项目把商户经纬度写入 Redis GEO：

```text
shop:geo:{typeId}
```

member 是商户 ID，score 是 Redis 内部编码后的地理位置。

查询时使用用户坐标：

```text
经度 x
纬度 y
商户类型 typeId
```

Redis 会根据经纬度计算距离，并返回附近商户 ID。

然后项目再根据这些商户 ID 去 MySQL 查询完整商户信息。

### 4.2 为什么 GEO 里只存商户 ID，不存商户完整信息？

Redis GEO 的核心作用是做地理位置计算。

它适合存：

```text
商户 ID + 经纬度
```

不适合存完整商户信息。

完整商户信息还是放在 MySQL 或普通 Redis 缓存里。

这样职责更清晰：

- Redis GEO：负责距离计算
- MySQL/缓存：负责商户详情

## 5. 点赞、关注、Feed 流

### 5.1 博客点赞为什么用 Redis Set？

博客点赞需要判断：

```text
当前用户是否点过赞
```

Set 天然适合保存不重复的用户 ID。

Key：

```text
blog:liked:{blogId}
```

Value：

```text
userId 集合
```

点赞时：

```text
SADD blog:liked:{blogId} userId
```

取消点赞时：

```text
SREM blog:liked:{blogId} userId
```

判断是否点赞：

```text
SISMEMBER blog:liked:{blogId} userId
```

### 5.2 Feed 流为什么用 ZSet？

关注推送需要按照时间排序。

ZSet 有两个部分：

- member：博客 ID
- score：发布时间戳

Key：

```text
feed:{userId}
```

当博主发布博客时，把博客 ID 推送给所有粉丝的 Feed 收件箱。

查询时按 score 倒序滚动分页。

### 5.3 Feed 流为什么不用普通分页？

普通分页：

```sql
limit offset, size
```

在数据持续新增时容易出现重复或漏数据。

Feed 流更适合滚动分页：

```text
maxTime + offset
```

其中：

- `maxTime` 表示本次查询的最大时间戳
- `offset` 处理同一时间戳下的多条数据

这样可以避免因为新数据插入导致分页错乱。

## 6. Bitmap 签到

### 6.1 签到为什么用 Bitmap？

签到只有两种状态：

```text
签了
没签
```

非常适合用 bit 表示。

一个月最多 31 天，只需要 31 个 bit。

Key：

```text
sign:{userId}:{yyyyMM}
```

第几天签到，就把对应 bit 设置为 1。

### 6.2 连续签到怎么统计？

使用 Redis 的 `BITFIELD` 取出当月到今天为止的签到数据。

然后从低位开始判断：

```text
最低位代表今天
如果是 1，连续签到天数 +1
如果是 0，说明连续签到中断
```

## 7. 滑动窗口限流

### 7.1 为什么要做限流？

秒杀接口容易被用户高频刷新，或者被脚本恶意请求。

如果不限制，会造成：

- Redis 压力升高
- MQ 消息堆积
- 后端线程被打满
- 正常用户体验变差

所以项目使用滑动窗口限流保护秒杀接口。

### 7.2 项目的限流是怎么实现的？

秒杀接口上加了：

```java
@RateLimit(maxRequests = 5, windowSeconds = 1)
```

含义：

```text
同一个用户 1 秒最多请求 5 次
```

底层实现：

```text
AOP 拦截接口
        |
构造限流 Key
        |
执行 rate_limit.lua
        |
Redis ZSet 删除窗口外请求
        |
统计窗口内请求数量
        |
超过阈值则拒绝
```

### 7.3 为什么滑动窗口用 ZSet？

ZSet 可以同时保存：

- 请求唯一 ID
- 请求时间戳 score

每次请求进来：

```text
删除 now - window 之前的数据
统计剩余数量
如果数量超过限制，拒绝
否则加入当前请求
```

这样可以精确统计任意时间窗口内的请求数。

### 7.4 为什么限流要用 Lua？

限流包含多个 Redis 操作：

```text
删除过期请求
统计当前请求数
判断是否超过阈值
写入当前请求
设置过期时间
```

如果这些命令分多次执行，在高并发下可能出现并发穿插。

Lua 在 Redis 中原子执行，可以保证整个限流判断过程不会被其他请求打断。

## 8. 秒杀异步下单

### 8.1 为什么秒杀不能直接写 MySQL？

如果秒杀请求都直接操作 MySQL：

```text
查库存
查是否买过
扣库存
插入订单
```

高并发下 MySQL 压力会非常大。

项目改成：

```text
Redis Lua 快速判断资格
        |
RocketMQ 异步写订单
```

这样可以把大量请求挡在 Redis 层，减少 MySQL 压力。

### 8.2 秒杀完整流程是什么？

```text
用户请求秒杀接口
        |
滑动窗口限流
        |
获取当前登录用户
        |
RedisIdWorker 生成订单 ID
        |
执行 seckill.lua
        |
Redis 判断库存和一人一单
        |
Redis 预扣库存
        |
Redis 记录用户已抢
        |
发送 RocketMQ 下单消息
        |
发送 RocketMQ 延迟关单消息
        |
返回 orderId
        |
消费者异步写 MySQL 订单
```

### 8.3 seckill.lua 解决了什么问题？

Lua 脚本解决了秒杀资格判断的原子性问题。

脚本中完成：

- 判断 Redis 库存是否存在且大于 0
- 判断用户是否已经抢过
- 扣减 Redis 库存
- 把用户加入已购集合

这些操作在 Redis 中一次性执行，不会被其他命令插入。

### 8.4 为什么接口返回 orderId 时 MySQL 里可能还没有订单？

因为项目使用异步下单。

接口返回 orderId 只说明：

```text
Redis 抢购资格成功
MQ 消息发送成功
```

不代表：

```text
MySQL 订单已经创建成功
```

订单真正创建是在 MQ 消费者中异步完成的。

如果要做得更完整，可以提供订单状态查询接口，让前端轮询订单是否创建完成。

### 8.5 MQ 发送失败怎么办？

可能出现：

```text
Redis 已经扣库存
但是 RocketMQ 消息发送失败
```

如果不处理，会导致：

```text
Redis 库存少了
用户被记录为已抢
但是 MySQL 没有订单
```

项目中的解决方式：

```text
捕获 MQ 发送异常
        |
回滚 Redis 库存
        |
从已购集合中删除 userId
        |
返回系统繁忙
```

### 8.6 Redis 库存和 MySQL 库存不一致怎么办？

异步系统中短暂不一致是正常的。

项目采用最终一致性：

- 秒杀入口先扣 Redis 库存
- MQ 消费时再扣 MySQL 库存
- MQ 发送失败则回滚 Redis
- MQ 消费失败则抛异常重试
- 超时未支付则回滚 Redis 和 MySQL

最终目标是：

```text
订单处理完成后，Redis 和 MySQL 状态保持一致
```

## 9. RocketMQ 消费幂等和失败重试

### 9.1 什么是 MQ 消费幂等？

幂等是指：

```text
同一个操作执行一次和执行多次，最终结果一样
```

在秒杀下单中就是：

```text
同一条 MQ 消息重复投递
不能重复扣库存
不能重复插入订单
```

### 9.2 为什么 RocketMQ 消息可能重复消费？

RocketMQ 通常保证的是至少一次投递。

可能出现：

```text
消费者已经处理成功
但是 ACK 失败
RocketMQ 以为消费失败
于是重新投递
```

所以业务必须自己做幂等。

### 9.3 项目怎么处理重复消费？

项目中有两层判断：

第一层，按 orderId 判断：

```java
VoucherOrder oldOrder = getById(voucherOrder.getId());
if (oldOrder != null) {
    return;
}
```

如果订单 ID 已经存在，说明这条消息之前处理过，直接返回。

第二层，按 userId + voucherId 判断：

```java
int count = query()
        .eq("user_id", userId)
        .eq("voucher_id", voucherOrder.getVoucherId())
        .count();
if (count > 0) {
    return;
}
```

如果用户已经买过这张券，也直接返回。

数据库再加唯一索引兜底：

```sql
ALTER TABLE tb_voucher_order
ADD UNIQUE KEY uk_user_voucher (user_id, voucher_id);
```

### 9.4 哪些情况应该 return，哪些情况应该 throw？

这是 MQ 消费中非常重要的问题。

应该 return 的情况：

```text
重复消息
重复下单
无效且不可恢复的消息
```

这些情况重试也没有意义。

应该 throw 的情况：

```text
数据库扣库存失败
保存订单失败
获取锁失败
数据库连接异常
```

这些属于本次消费没有真正完成，需要 RocketMQ 后续重试。

总结：

```text
重复消息 return
真正失败 throw
```

### 9.5 如果 catch 住异常但不抛出，会发生什么？

RocketMQ 会认为：

```text
消费者方法正常执行结束
消息消费成功
```

这会导致消息丢失。

例如：

```text
MySQL 订单没有保存成功
但是异常被 catch 后只打印日志
方法正常 return
MQ 就不会再投递
```

所以真正失败时必须抛异常。

## 10. 超时关单和支付并发

### 10.1 为什么需要超时关单？

用户抢到优惠券后，如果一直不支付，会长期占用库存。

所以项目下单成功后发送 RocketMQ 延迟消息。

延迟时间：

```text
15 分钟
```

如果 15 分钟后订单仍未支付，则自动取消订单并释放库存。

### 10.2 超时关单流程是什么？

```text
RocketMQ 延迟消息到达
        |
OrderTimeoutListener 收到 orderId
        |
获取订单维度 Redisson 锁
        |
查询订单
        |
判断订单是否仍然未支付
        |
修改订单为已取消
        |
MySQL 库存 +1
        |
Redis 库存 +1
        |
Redis 已购集合删除 userId
```

### 10.3 支付回调和超时关单同时发生怎么办？

这是面试高频问题。

可能出现：

```text
用户刚好支付成功
延迟关单消息也刚好到了
```

如果不控制，可能出现：

```text
支付线程把订单改成已支付
关单线程把订单改成已取消
```

项目中的解决方案：

```text
支付回调和超时关单都使用同一把订单锁
```

锁：

```text
lock:order:{orderId}
```

同时 SQL 更新时带状态条件：

```text
只有未支付订单才能改成已支付
只有未支付订单才能改成已取消
```

所以最终只有一个操作能成功。

## 11. Redisson 分布式锁

### 11.1 为什么要用分布式锁？

在单机环境下，Java 的 `synchronized` 可以控制并发。

但是项目如果部署多台机器：

```text
机器 A 一个 JVM
机器 B 一个 JVM
```

`synchronized` 只能锁住当前 JVM，不能锁住其他机器。

分布式锁通过 Redis 这类外部中间件实现，所有服务实例都去竞争同一把锁。

### 11.2 Redisson 加锁的基本原理是什么？

Redisson 锁底层主要使用 Redis Hash。

Key 类似：

```text
order:1001
```

Hash field 通常包含：

```text
客户端 ID + 线程 ID
```

Value 是重入次数。

加锁时：

```text
如果锁不存在，创建锁，value = 1
如果锁已经存在，但持有者是当前线程，value +1
如果锁存在且不是当前线程，获取锁失败
```

这就是可重入锁的基础。

### 11.3 什么是可重入锁？

可重入是指：

```text
同一个线程已经拿到锁后，可以再次拿同一把锁
```

例如：

```java
lock.lock();
try {
    methodA();
} finally {
    lock.unlock();
}
```

如果 `methodA()` 内部又尝试获取同一把锁，只要是同一个线程，就应该允许成功。

Redisson 通过 Hash 中的计数实现可重入：

```text
第一次加锁 value = 1
第二次重入 value = 2
释放一次 value = 1
再释放一次才真正删除锁
```

### 11.4 Redisson 为什么用 Lua 脚本？

加锁和释放锁都不是单条 Redis 命令能完整表达的。

例如释放锁需要：

```text
判断锁是否属于当前线程
如果不是，不能删除
如果是，重入次数 -1
如果次数为 0，删除锁
发布释放锁消息
```

这些操作必须原子执行。

Lua 脚本可以保证多个 Redis 命令作为一个整体执行，不会被其他命令插入。

### 11.5 为什么释放锁必须判断是不是自己的锁？

如果不判断锁归属，可能出现误删别人锁。

例如：

```text
线程 A 获取锁，业务超时，锁自动过期
线程 B 获取到同一把锁
线程 A 执行完后调用 unlock
如果不判断归属，就会把线程 B 的锁删掉
```

Redisson 释放锁时会判断：

```text
当前线程是否是锁持有者
```

只有自己的锁才能释放。

### 11.6 Redisson 的 WatchDog 是什么？

WatchDog 是 Redisson 的看门狗续期机制。

如果加锁时没有手动指定锁过期时间，Redisson 默认会给锁设置 30 秒过期时间，并启动后台续期任务。

只要持锁线程还活着，WatchDog 会定期续期。

这样可以避免：

```text
业务执行超过锁过期时间
锁提前释放
其他线程进来并发执行
```

### 11.7 如果指定了 leaseTime，还会有 WatchDog 吗？

通常不会。

例如：

```java
lock.tryLock(1, 30, TimeUnit.SECONDS)
```

这里指定了锁最多持有 30 秒，Redisson 会认为你自己明确指定了过期时间，不再自动续期。

所以使用时要注意：

- 业务时间不确定：可以不指定 leaseTime，让 WatchDog 续期
- 业务时间可控：可以指定 leaseTime，避免锁长期不释放

### 11.8 Redisson 获取锁失败后如何等待？

Redisson 内部使用 Redis Pub/Sub 机制。

当线程获取锁失败时，会订阅锁释放消息。

释放锁时，持锁线程会发布消息。

等待线程收到消息后，再重新尝试获取锁。

这样比一直 while 循环重试更高效。

### 11.9 Redisson 分布式锁在项目中用在哪里？

主要有两个场景：

1. 秒杀订单消费时，使用用户维度锁：

```text
order:{userId}
```

作用：

```text
同一个用户的订单消息同一时间只处理一个
```

2. 支付回调和超时关单时，使用订单维度锁：

```text
lock:order:{orderId}
```

作用：

```text
防止支付和关单同时修改同一笔订单状态
```

### 11.10 Redisson 锁能完全保证业务安全吗？

不能只依赖锁。

原因：

- 锁可能因为网络问题获取失败
- 业务可能异常
- 多系统之间还有状态一致性问题

所以项目中还配合了：

- Lua 原子判断
- 数据库唯一索引
- 状态条件更新
- MQ 幂等消费
- 失败重试

锁只是并发控制的一部分，不是全部。

## 12. 数据库唯一索引

### 12.1 为什么已经有 Java 判断，还要加唯一索引？

Java 判断：

```java
count user_id + voucher_id
```

在高并发下可能出现：

```text
线程 A 查 count = 0
线程 B 查 count = 0
线程 A 插入成功
线程 B 也插入成功
```

数据库唯一索引是最后一道保险。

即使 Java 判断漏了，数据库也会拒绝重复数据。

### 12.2 唯一索引含义是什么？

```sql
ALTER TABLE tb_voucher_order
ADD UNIQUE KEY uk_user_voucher (user_id, voucher_id);
```

含义：

```text
同一个 user_id + voucher_id 组合只能出现一次
```

允许：

```text
用户 A 买券 1
用户 A 买券 2
用户 B 买券 1
```

不允许：

```text
用户 A 重复买券 1
```

## 13. Redis 全局 ID 生成器

### 13.1 为什么不用数据库自增 ID？

秒杀场景下订单量可能很大，如果依赖数据库自增 ID：

- 所有请求都要访问数据库
- 分库分表后自增 ID 容易冲突
- 不适合高并发场景

项目使用 Redis 自增序列 + 时间戳生成全局 ID。

### 13.2 ID 生成器的组成是什么？

大致组成：

```text
符号位 + 时间戳 + 序列号
```

每天 Redis 维护一个自增 key：

```text
icr:order:yyyy:MM:dd
```

例如：

```text
icr:order:2026:06:29 -> 30000
```

这个值只是当天序列号，不是最终订单 ID。

最终订单 ID 是：

```text
时间戳和序列号通过位运算拼接得到
```

## 14. 常见压测问题

### 14.1 JMeter 多线程是否等于多用户？

不一定。

如果所有线程使用同一个 token：

```text
1000 个线程 = 同一个用户请求 1000 次
```

如果 CSV 中每一行是不同 token，并且请求头使用：

```text
Authorization: ${token}
```

才表示多个不同用户。

### 14.2 为什么 1000 个请求异常率很高？

可能不是系统崩了，而是业务规则生效。

例如：

```text
库存只有 100
1000 个用户来抢
只有 100 个成功
剩下 900 个库存不足
```

JMeter 如果设置了断言，可能把这些业务失败也统计为异常。

### 14.3 压测秒杀应该看什么指标？

重点看：

- 成功订单数是否等于库存
- MySQL 库存是否为 0
- Redis 库存是否为 0
- 是否有重复订单
- 是否有超卖
- MQ 是否有消息堆积
- 接口平均响应时间和 99 线响应时间

## 15. 面试高频追问

### 15.1 如果 Redis 扣库存成功，但 MQ 发送失败怎么办？

项目中会回滚 Redis：

```text
seckill:stock:{voucherId} +1
seckill:order:{voucherId} 删除 userId
```

然后返回系统繁忙。

### 15.2 如果 MQ 消费失败怎么办？

消费者抛异常。

RocketMQ 感知到消费失败后会重新投递消息。

### 15.3 如果 MQ 重复投递怎么办？

项目中通过：

- orderId 判重
- userId + voucherId 判重
- 数据库唯一索引兜底

保证重复消息不会重复下单。

### 15.4 如果支付成功和超时关单同时发生怎么办？

使用同一把订单维度 Redisson 锁：

```text
lock:order:{orderId}
```

并且数据库更新时带订单状态条件：

```text
只有未支付订单才能支付成功
只有未支付订单才能关闭
```

### 15.5 如果订单超时消息到了，但订单还没落库怎么办？

这可能发生在异步系统中：

```text
延迟消息先被消费
普通订单消息还没成功落库
```

项目中将这种情况视为可重试异常，抛出异常让 RocketMQ 后续重试。

### 15.6 如果 Redis 和 MySQL 库存短暂不一致怎么办？

这是异步系统的正常现象。

项目通过 MQ 重试、Redis 回滚、超时关单回滚来保证最终一致。

### 15.7 为什么不用 synchronized？

`synchronized` 只能锁当前 JVM。

项目如果部署多个实例，不同实例之间无法共享锁。

Redisson 分布式锁基于 Redis，所有服务实例竞争同一把锁，可以跨 JVM、跨机器生效。

### 15.8 Redisson 锁和自己写 Redis 锁有什么区别？

自己写 Redis 锁通常只处理：

```text
SET key value NX EX seconds
```

但还要自己处理：

- 释放锁原子性
- 锁误删问题
- 可重入
- WatchDog 自动续期
- 锁等待和唤醒
- 公平锁、读写锁等扩展能力

Redisson 已经封装了这些复杂逻辑。

### 15.9 为什么限流不用固定窗口？

固定窗口在窗口边界可能出现流量突刺。

例如：

```text
第 1 秒最后 100ms 进来 5 次
第 2 秒最开始 100ms 又进来 5 次
```

短时间内实际请求达到 10 次。

滑动窗口按当前时间向前计算，更平滑、更准确。

### 15.10 这个项目还有哪些可以继续优化？

可以继续优化：

- 增加订单状态查询接口
- 增加 RocketMQ 死信队列处理
- 增加消息发送失败的本地事务表或 Outbox 机制
- 增加真实支付签名校验
- 增加 Prometheus + Grafana 监控
- 增加单元测试和集成测试
- 将配置拆分成 application-local.yaml
- 前端目录结构进一步整理

## 16. 面试回答模板

### 16.1 如何介绍秒杀系统？

可以这样回答：

```text
我这个项目的秒杀不是直接写 MySQL，而是先通过 Redis Lua 脚本完成库存校验、一人一单校验和 Redis 预扣库存。Lua 在 Redis 中原子执行，可以避免并发超卖。Lua 成功后生成订单 ID，并发送 RocketMQ 消息异步落库，接口直接返回 orderId，从而降低接口响应时间。

消费者收到消息后，会先加用户维度 Redisson 锁，再通过 orderId 判断消息是否已经处理过，通过 userId + voucherId 判断用户是否已经买过，并在数据库层加了 user_id、voucher_id 唯一索引兜底，避免 MQ 重复投递导致重复下单。如果 MySQL 扣库存或保存订单失败，会抛异常让 RocketMQ 重试。

下单成功后还会发送 RocketMQ 延迟消息，如果订单 15 分钟内未支付，就自动关单，并回滚 MySQL 和 Redis 库存。支付回调和超时关单使用同一把订单维度 Redisson 锁，防止并发修改订单状态。秒杀接口外层还加了基于 Redis ZSet + Lua 的滑动窗口限流，防止用户恶意高频请求。
```

### 16.2 如何介绍 Redisson 锁？

可以这样回答：

```text
项目中使用 Redisson 分布式锁解决跨 JVM 的并发问题。Redisson 锁底层主要使用 Redis Hash 结构，field 是客户端 ID 和线程 ID，value 是重入次数，因此支持可重入。加锁、解锁等复杂逻辑通过 Lua 脚本保证原子性。释放锁时会先判断当前线程是否是锁持有者，避免误删其他线程的锁。

如果加锁时不指定 leaseTime，Redisson 会启动 WatchDog 看门狗机制，默认给锁设置 30 秒过期时间，并在业务执行期间自动续期，避免业务没执行完锁就过期。获取锁失败时，Redisson 会利用 Redis Pub/Sub 订阅锁释放消息，避免一直空转重试。

在本项目中，秒杀订单消费使用用户维度锁防止同一用户并发下单；支付回调和超时关单使用订单维度锁，保证同一笔订单状态不会被并发修改。
```
